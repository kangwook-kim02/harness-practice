# Architecture Decision Records

## 철학
MVP 속도 최우선. 외부 의존성 최소화. 작동하는 최소 구현을 선택.

---

### ADR-001: Next.js 15 App Router 선택
**결정**: 프론트엔드와 API 서버 모두 Next.js App Router로 구현  
**이유**: 프론트+백을 하나의 프로젝트로 관리하여 복잡도 감소. Server Components로 초기 렌더링 성능 확보.  
**트레이드오프**: 순수 React SPA 대비 학습 곡선이 있으나, 별도 백엔드 서버 운영 불필요

### ADR-002: Supabase 선택
**결정**: DB(PostgreSQL), Auth, Storage를 Supabase로 통합  
**이유**: 인증·DB·파일 저장 인프라를 한 번에 해결하여 MVP 개발 속도 극대화. 무료 플랜으로 초기 운영 가능.  
**트레이드오프**: Supabase 플랫폼 의존성이 생기지만, 직접 PostgreSQL 운영 대비 초기 설정 비용이 대폭 줄어듦

### ADR-003: 관리자 수동 승인 방식
**결정**: 학생 인증을 OCR 자동화 없이 관리자가 직접 검토 후 승인  
**이유**: OCR 구현 복잡도 및 오인식 위험 제거. 관리자 페이지 추가만으로 구현 가능.  
**트레이드오프**: 승인까지 대기 시간이 발생하지만, 인증 신뢰도가 높아지고 구현이 단순함

### ADR-004: 연락처 직접 노출 방식
**결정**: 메시지 기능 대신 매물 상세 페이지에 등록자 연락처를 직접 표시  
**이유**: MVP 범위 최소화. 메시지 시스템(conversations/messages 테이블, 실시간 처리)은 구현 복잡도가 높아 제외.  
**트레이드오프**: 연락처가 노출되어 스팸 위험이 있지만, 인증된 사용자만 등록 가능하므로 최소한의 신뢰 확보
